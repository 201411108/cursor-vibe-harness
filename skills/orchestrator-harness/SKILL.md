# Orchestrator Harness

사용자 요청을 분석하고, 적절한 역할 스킬을 순차 호출하여 최적의 결과를 만드는 오케스트레이터.

## 실행 프로토콜

### Phase 0: 작업 분석

사용자 요청을 받으면:

1. `.cursor/specs/` 폴더 존재 여부를 확인한다 (Glob으로 `.cursor/specs/features/*.md` 탐색)
2. 관련 기획 문서가 있으면 Read로 읽어 컨텍스트를 확보한다
3. 아래 기준으로 요청을 분류한다:

| 작업 유형 | 호출 역할 | 설명 |
|-----------|-----------|------|
| 신규 기능 개발 | 기획자 -> 디자이너 -> 개발자 | 기획 분석 + UI 검토 + 구현 |
| UI/UX 개선 | 디자이너 -> 개발자 | 디자인 분석 + 구현 |
| 기획 검토/PRD | 기획자 | 기획 분석만 |
| 버그 수정 | 개발자 | 즉시 구현 |
| 리팩토링 | 개발자 | 즉시 구현 |
| 디자인 리뷰 | 디자이너 | 디자인 분석만 |
| 복합 요청 | 기획자 -> 디자이너 -> 개발자 | 전체 파이프라인 |

**판단 기준**:
- 요구사항이 모호하거나 엣지 케이스가 불분명하면 -> 기획자 포함
- UI 변경이 포함되어 있으면 -> 디자이너 포함
- 코드 변경이 필요하면 -> 개발자 포함
- 피그마 URL이 첨부되어 있으면 -> 반드시 디자이너 포함

### Phase 1: 역할 선정 선언

분석 결과를 사용자에게 간결하게 선언한다:

```
## 작업 분석 결과

**작업 유형**: [유형]
**활성 역할**: [역할1] -> [역할2] -> ...
**근거**: [왜 이 역할들이 필요한지 1줄 설명]
```

### Phase 2: 순차 실행

선정된 역할을 순서대로 실행한다. 각 역할 실행 시:

1. 해당 역할의 SKILL.md를 **Read 도구로 읽는다**
2. `.cursor/specs/`에서 관련 기획 문서를 참조한다 (있는 경우)
3. SKILL.md의 지시에 따라 작업을 수행한다
4. 산출물을 **컨텍스트 블록** 형식으로 정리한다
5. 산출물을 `.cursor/specs/`에 기록한다 (문서 관리 프로토콜 참조)
6. 다음 역할 실행 시 이전 산출물을 참조한다

### Phase 3: 최종 정리

모든 역할 실행이 완료되면:
- 각 역할의 핵심 산출물을 요약한다
- 남은 질문이나 후속 작업이 있으면 명시한다
- `.cursor/specs/` 문서 갱신 여부를 확인한다:
  - 기획 문서가 생성/갱신되었는가?
  - 주요 결정사항이 기록되었는가?
  - 변경 이력이 남겨졌는가?

## 컨텍스트 전달 프로토콜

각 역할의 산출물은 다음 형식으로 정리하여 이후 역할이 참조한다:

```markdown
---
### [역할명] 산출물

**요약**: [1-2줄 핵심 요약]

**상세 내용**:
[역할별 산출물 본문]

**다음 역할에 전달할 핵심 사항**:
- [항목 1]
- [항목 2]
---
```

## 역할 스킬 경로

| 역할 | 스킬 경로 |
|------|-----------|
| 기획자 | `~/.cursor/skills/role-planner/SKILL.md` |
| 디자이너 | `~/.cursor/skills/role-designer/SKILL.md` |
| 개발자 | `~/.cursor/skills/role-developer/SKILL.md` |

## 모델 추천 (Task subagent 사용 시)

순차 실행이 기본이지만, 사용자가 병렬 실행을 요청하거나 작업 규모가 클 때 Task subagent를 활용할 수 있다:

| 역할 | 추천 모델 | 이유 |
|------|-----------|------|
| 기획자 | default | 깊은 분석과 추론 필요 |
| 디자이너 | default | 시각적 판단과 UX 추론 필요 |
| 개발자 (단순) | fast | 명확한 구현, 빠른 실행 |
| 개발자 (복잡) | default | 아키텍처 결정, 다파일 변경 |

## 문서 관리 프로토콜

모든 작업에서 `.cursor/specs/` 폴더를 Single Source of Truth로 활용한다:

| 시점 | 행동 | 대상 폴더 |
|------|------|-----------|
| 작업 시작 전 | 관련 기획 문서를 탐색하고 참조 | `features/` |
| 기획자 완료 후 | 기획 산출물을 저장/갱신 | `features/{feature-name}.md` |
| 디자이너 완료 후 | 디자인 결정사항 기록 | `decisions/{NNN}-{title}.md` |
| 개발자 완료 후 | 변경 이력 기록 + 기획 문서 갱신 | `changes/{date}-{title}.md` + `features/` |
| 유지보수 작업 시 | 기존 문서를 먼저 읽고 변경분 반영 | 해당 문서 |

`.cursor/specs/` 폴더가 없으면 문서 기록을 생략하되, 사용자에게 `npx cursor-agent-harness init`으로 초기화할 수 있음을 안내한다.

## 바이패스 조건

아래 경우에는 오케스트레이터를 거치지 않고 직접 처리한다:
- 단순 질문 (코드 설명, 문법 질문 등)
- 단일 파일의 사소한 수정
- git 명령어 실행
- 파일 읽기/검색만 필요한 경우
- 명시적으로 특정 역할만 요청한 경우
